# 商用販売に向けた残課題

document-agent を「販売」する際の懸念点を洗い出した結果のうち、コードだけで安全に
対処できる項目は本ブランチで対応済み。本ドキュメントは **未対応の課題** を優先度別に
記録する。背景は `docs/v2-design.md`、セキュリティモデルは `SECURITY.md` を参照。

## 対応済み（本ブランチ）

- 本番設定ガード（`lib/config-guard.ts` / `instrumentation.ts`） — 本番で「認証オフ」
  「MCP allowlist 未設定」「デモモード ON」をフェイルセキュアに拒否
- 監査ログ基盤（`lib/audit-log.ts`） — 認証・検索・編集・PR 作成の構造化ログ
- テストスイート（Vitest、`tests/`） — 主要 lib モジュールのユニットテスト
- 運用ドキュメント（`SECURITY.md`）と LICENSE（プロプライエタリ）

---

## 残課題 — 販売前に必須（ブロッカー）

### 1. マルチテナント分離（SaaS 外販する場合のみ）
現状は「1 組織 = 1 デプロイ」前提。`tenant_id` の概念がなく、1 インスタンスを複数顧客で
共有すると全顧客が全文書を閲覧できてしまう。SaaS 外販するならテナント分離は再設計規模。
→ オンプレ／個別デプロイ販売なら不要（§販売形態を参照）。

### 2. LLM ベンダー選定・データ越境方針
検索・編集で文書本文が Google Gemini API に送信される。機密文書（人事考課・ISMS 等）を
本番運用するなら、データ学習対象になりうる無料枠は不可。Gemini 有料枠 or 国内 DC の
さくらの AI Engine への切替、および顧客への外部送信の開示が必要（`docs/v2-design.md` §7）。

### 3. 保存時暗号化
文書は GitHub 上に平文 Markdown で保存。規制業界（金融・自治体・医療）向けには
保存時暗号化・鍵管理（CMEK 相当）が要求されることが多い。GitHub バックエンドのままでは
対応が難しく、保存先の見直しが必要になる可能性。

### 4. データ削除・GDPR 対応
GitHub の Git 履歴は不変。特定文書の完全消去・「忘れられる権利」に構造的に対応できない。
削除機構の設計が必要。

### 5. ユーザー単位の即時アクセス失効
権限は環境変数の allowlist 管理で、退職者の即時遮断に再デプロイが必要。
ユーザー管理・セッション失効の仕組みが要る（現行の「DB を持たない」原則と要調整）。

## 残課題 — 品質・スケール

### 6. next-auth の安定版化
現状 `next-auth@^5.0.0-beta.31`（ベータ）。認証基盤をベータのまま本番運用するのは
リスク。安定版リリース後に更新する。

### 7. 実データ規模での検証
メタデータ駆動検索は数十〜数百件のダミーコーパスを前提に設計。実顧客の大規模コーパスでの
検索精度・段階的開示の効きは未検証。5,000 件規模へのスケール方針は `docs/v2-design.md`
§12 を参照。

### 8. テストカバレッジの拡充
本ブランチのテストは純粋 lib モジュール中心。API ルートの統合テスト、E2E、
プロンプトインジェクションのレッドチームテストは未着手。

### 9. シークレット管理の強化
`GEMINI_API_KEY` / `GITHUB_TOKEN` / `AUTH_SECRET` は環境変数依存でローテーション機構なし。
`AUTH_SECRET` は HS256 対称鍵で漏洩時に JWT 偽造が可能。シークレットマネージャ導入と
ローテーション運用を検討する。

### 10. 4 段階 RBAC
現状は 2 ロール（`一般` / `編集`）。設計が掲げる viewer / proposer / approver / admin の
4 段階＋カテゴリ別権限は未実装（`docs/v2-design.md` §4-D）。

### 11. 顧客別セットアップの自動化
個別デプロイ販売では、GitHub チーム作成・CODEOWNERS・ブランチ保護の手作業（
`.github/branch-protection.md`）が顧客ごとに発生する。IaC（Terraform 等）での自動化が
ないと導入のたびに事故りやすく、サポート工数も膨らむ。

## 残課題 — 事業・法務

### 12. 依存ライブラリのライセンス棚卸し・NOTICE
主要依存は MIT / Apache 系で概ね商用可だが、`pdfjs-dist` 等は要確認。NOTICE ファイルの
整備が必要。

### 13. SLA・契約・プライバシーポリシー
Gemini / GitHub 障害時の可用性、誤った AI 編集提案による損害の責任分界点、外部サービス
経由でのデータ処理の明示などを契約・規約で詰める。

### 14. 監査ログの保全・改ざん耐性
現状の監査ログは標準出力。長期保全・改ざん検知（追記専用ストア等）は未対応。
コンプラ監査（SOC2 / ISMS 認証取得）を狙う場合に必要。

---

## 補足: 販売形態の選択（未決定）

懸念点の重み付けは販売形態で大きく変わる。現アーキテクチャ（GitHub バックエンド・
DB を持たない・CODEOWNERS = 組織のチーム・対象リポジトリを環境変数指定）は
**「1 組織 = 1 デプロイ」を前提**としており、現実的なスタートは
**オンプレ／個別デプロイ販売**。マルチテナント SaaS 外販を狙う場合は課題 1 の
テナント分離が前提となり、別プロジェクト規模の再設計が必要。
