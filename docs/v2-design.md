# v2 設計サマリ（実装引き継ぎ用）

このドキュメントは v2（ハイブリッド型ドキュメントエージェント）の設計意図と実装方針を、新しい実装セッションが**このファイルと AGENTS.md だけ読めば着手できる**形にまとめたものです。

---

## 1. 出発点

### 現状（旧アイディア = `legacy/v1-original` ブランチ）
- 50 件の社内 Markdown ドキュメント検索システム
- Google Gemini Flash / Flash-Lite による段階的検索（3 ステップ）
  1. フロントマター → 候補抽出（Flash-Lite）
  2. セクション本文取得（ファイル I/O）
  3. 回答生成（Flash）
- ベクトル DB なし、シンプルな Next.js 1 サービス構成
- 詳細：`README.md` 参照

### ゴール（v2）
旧アイディアの **段階的開示の優位性を維持** したまま、市販 SaaS（InfoCraft 等）相当の汎用性・拡張性・編集機能を取り込んだ「**いいとこ取りのドキュメントエージェント**」を構築する。

---

## 2. 設計の基本方針

| 原則 | 内容 |
|---|---|
| **メタデータ駆動が主、ベクトルが補助** | 段階的開示の構造を壊さない。ベクトル検索は Step1 の候補プール拡張用途 |
| **構造化編集指示 `{find, replace}`** | AI 生成の編集は必ず構造化、人間が diff レビュー前提 |
| **GitHub をバックエンドに据える** | branch = ドラフト、PR = 承認待ち、merge = 反映。CODEOWNERS でカテゴリ別承認者 |
| **小さく入れて積み重ねる** | フェーズ分けで段階導入。各フェーズ単独で価値が出る単位 |
| **LLM 抽象化を最小限残す** | モデル名・API キーを環境変数化、Sakura 等への切替余地を確保 |

### F. MCP コネクタ（リモート MCP サーバ）

| 項目 | 主要要素 | 確定度 |
|---|---|---|
| MCP コネクタ | 文書検索を MCP ツールとして公開。利用者が自分の Claude に登録 | 導入決定 |

提供レイヤー（§3 ④）に「リモート MCP サーバ」を追加する。開発者側が Gemini を呼んで回答生成する従来構成に対し、**検索 API のみをホスティングし、クエリ時の推論（候補の最終選定・回答生成）は利用者側の Claude が負担する**構成。開発者は LLM 推論コストを負わず、利用者は自分の契約・トークンで社内文書を検索・質問できる。

- **エンドポイント**: `app/api/mcp/route.ts`（Next.js ルートハンドラ）。公式 SDK `@modelcontextprotocol/sdk` + Streamable HTTP トランスポートで実装。
- **ツール**: `search_documents`（候補プール）/ `get_sections`（セクション本文）/ `list_categories`（カテゴリ一覧）/ `propose_edit`（構造化編集 `{find, replace, reason}[]` を逐語適用し branch + PR を作成）。`propose_edit` は MCP の input_schema で編集構造を強制（Gemini の responseSchema の代替）し、`mcp:edit` スコープを要求。逐語マッチに失敗した編集が 1 件でもあれば PR を作らず診断を返す。反映は GitHub の PR レビューで（自動マージなし、§10 整合）。
- **段階的開示の維持**: サーバはクエリ時に回答生成 LLM を呼ばない。各ツールが返す情報量を絞る（フロントマター → セクション本文）ことで §3 の段階的開示構造をツール境界上で保つ。
- **ベクトル検索**: クエリ埋め込みのみ Gemini を使用。`GEMINI_API_KEY` 未設定時はメタデータ駆動のみへ自動フォールバック（既存 `lib/hybrid-search.ts` の挙動を流用）。
- **疎結合**: ツール実装（`lib/mcp/`）は既存の `lib/` モジュール（`document-utils` / `hybrid-search` / `edit-schema` / `github`）を import するだけにとどめ、既存の Gemini チャットパイプラインと共存させる。
- **認証（OAuth 2.1）**: `/api/mcp` は OAuth 2.1 のリソースサーバとして保護されると同時に、最小限の認可サーバも兼ねる。クライアント（claude.ai のカスタムコネクタ等）は動的クライアント登録 → authorization code + PKCE(S256) フローでトークンを取得する。実ユーザー認証は既存 `auth.ts`（NextAuth + Google OIDC）に委譲し、社内ユーザーのみを allowlist（`MCP_ALLOWED_EMAIL_DOMAINS` / `MCP_ALLOWED_EMAILS`）で許可する。アクセストークン・リフレッシュトークン・認可コード・client_id はすべて `AUTH_SECRET` で署名した HS256 JWT で表現し、DB を使わない（§10 整合。認可コードの単回使用のみプロセス内メモリでベストエフォート保証）。エンドポイント: `/.well-known/oauth-protected-resource`・`/.well-known/oauth-authorization-server`・`/api/mcp/oauth/{register,authorize,token}`。アプリ全体の認証が OFF の開発時はトークン不要（Phase 1 と同じ挙動）。認証が有効になるまで公開デプロイしない。

## 5. 実装フェーズと優先順位

| 順 | 項目 | 所要目安 | 理由 |
|---|---|---|---|
| 1 | 多言語対応 | 1 日 | プロンプト変更のみ、即効性 |
| 2 | プロンプトキャッシュ（暗黙的の観測ログ）| 1 日 | ヒット率測定から始める |
| 3 | 多形式取り込み + フロントマター自動生成 | 3〜5 日 | 取り込みパイプラインがハイブリッド検索の前提 |
| 4 | ハイブリッド検索（軽量版）| 2〜3 日 | `embeddings.json` + JS でコサイン類似度（pgvector 不要）|
| 5 | GitHub バックエンド化 | 3〜5 日 | 編集機能の前提 |
| 6 | チャットベース編集 Phase 1（PoC）| 1 週間 | 認証なし・ローカルで体験検証 |
| 7 | 権限管理 + 承認フロー | 1 週間 | Auth.js + GitHub PR フロー |
| 8 | 明示的プロンプトキャッシュ | 1〜2 日 | アクセス増えてから |
| 9 | 埋め込み配布 / 分析ダッシュボード | 1〜2 週間 | 外販 / 運用拡張時 |

各フェーズは**単独で価値が出る単位**に切ること。途中で止めても旧アイディアより改善された状態を維持する。

## 6. 技術スタック決定事項

| 領域 | 採用 | 理由 |
|---|---|---|
| **Framework** | Next.js 16（既存維持）| Cloudflare Workers（OpenNext）デプロイ、breaking changes は AGENTS.md 参照 |
| **LLM（開発フェーズ）** | Gemini 2.5 Flash + Flash-Lite | 無料枠 45,000 req/月、長文コンテキスト、既存実装 |
| **LLM（本番フェーズ判断）** | Gemini 有料 or さくらの AI Engine | コンプラ要件次第。**LLM 設定は環境変数で抽象化** |
| **Embedding** | Gemini gemini-embedding-001（暫定）| 後にさくらの multilingual-e5-large 検討 |
| **ベクトル保存（初期）** | `documents/embeddings.json` + JS でコサイン計算 | 50〜数百件規模では pgvector 不要 |
| **ベクトル保存（拡張）** | pgvector（Supabase）| 500 件超で移行 |
| **差分エディタ** | `@monaco-editor/react`（DiffEditor）| VS Code 同等、左右編集可 |
| **編集指示生成** | Gemini `responseSchema` で `{find, replace, reason}[]` 強制 | 型安全な構造化出力 |
| **永続化** | GitHub（Octokit）| PR 承認 = 編集承認、CODEOWNERS = カテゴリ別承認者 |
| **認証（Phase 2 以降）** | Auth.js（NextAuth）+ Google OIDC | 社内利用想定 |
| **同時編集制御** | 楽観ロック（branch 名 + version）| Git の merge 機構を活用 |

## 7. LLM 戦略（開発 → 本番）

### 開発フェーズ（現在）
- **Gemini 無料枠で開始**：ダミーデータ前提、データ学習利用問題なし
- **抽象化レイヤを薄く張る**：
  ```ts
  // lib/llm-config.ts（実装時に作成）
  export const llmConfig = {
    candidateModel: process.env.LLM_CANDIDATE_MODEL ?? 'gemini-2.5-flash-lite',
    answerModel: process.env.LLM_ANSWER_MODEL ?? 'gemini-2.5-flash',
    embeddingModel: process.env.LLM_EMBEDDING_MODEL ?? 'text-embedding-004',
    apiKey: process.env.GEMINI_API_KEY!,
  };
  ```
- これでモデル切替・本番移行時にコード変更を最小化

### 本番フェーズ（移行判断）
以下のいずれかが該当した場合は**さくらの AI Engine（国内 DC）も候補**：
- 機密ドキュメント（人事考課・ISMS 等）を本番運用
- 顧客から国内処理証明を求められる
- 規制業界（金融・自治体・医療）向け展開

該当しない場合は **Gemini 有料へのキー差替えのみ**で本番移行。

## 10. やらないことリスト（重要）

設計の振れを防ぐため、明示的に**現時点で採用しない**選択肢：

- ❌ ベクトル DB を最初から導入（pgvector / Pinecone 等）→ 件数増加時に移行
- ❌ 全文再生成型の AI 編集（必ず `{find, replace}` 構造化）
- ❌ 自動マージ（AI 提案 → 人間レビュー必須）
- ❌ ChatGPT API への乗り換え（コスト効率で Gemini 優位、変える理由なし）
- ❌ 無料 OSS LLM への置換（現規模では不要、品質劣化リスクが上回る）
- ❌ DB（Postgres 等）で文書本体を管理（GitHub バックエンドが第一選択）

## 11. 参考：旧アイディアと InfoCraft の比較

旧アイディアが InfoCraft に勝っている点（v2 でも引き継ぐべき優位性）：

1. **段階的開示によるトークン効率**（8 万 → 1.5 万 tokens）
2. **フロントマター駆動でハルシネーション抑制**
3. **軽量/高性能モデルの使い分け**（Flash-Lite + Flash）
4. **セクション単位の精密な引用**
5. **ベクトル DB 不要の運用シンプルさ**
6. **更新が爆速（再 Embedding 不要）**
7. **JSON 出力でデバッグ可能性が高い**

これらを **「メタデータ駆動を主、ベクトル補助」** の原則で守りながら、InfoCraft の汎用性（多形式・多言語・編集機能）を足すのが v2 の本質。

## 12. 大規模化（5,000 件規模・エンタープライズ移行）への展望

§5 の現行フェーズは数十〜数百件のコーパスを前提にしている。大企業への導入では文書数が **5,000 件規模**（さらにその先）に達し、現状のままだと 2 つの移行障壁が表面化する。本節はその解決方針を示す。**この節は将来展望であり、現時点では未着手**。実装は §5 Phase 9 以降、または独立トラックとして段階導入する。

### 12.1 想定される 2 つの障壁

| 障壁 | 内容 |
|---|---|
| **既存文書の変換コスト** | 取り込みパイプライン（§4-B）は現状 1 ファイル＝1 LLM 呼び出しの直列実行（`scripts/ingest.mjs`）。数千件で非現実的な所要時間になり、「乗り換え準備に数週間」が導入障壁になる |
| **文書量による検索レイテンシ** | Step 1 が全文書のフロントマターをプロンプトに入れる（件数に線形）。ベクトル検索も全件線形コサイン（`lib/hybrid-search.ts`）。50 件で index.json は約 41 KB ≈ 約 1 万トークン。5,000 件では **約 100 万トークン規模**となり、軽量モデルのコンテキストを実質超過してコスト・レイテンシが破綻する |

### 12.2 障壁1の解決方針：変換と enrichment の分離

決定的・高速・**API 不要**な「変換」（多形式 → MD、セクションマーカー注入）と、レート律速で遅い「LLM enrichment（フロントマター生成）」を分離する。遅いのは後者だけ。

| 方針 | 内容 |
|---|---|
| **Day 1 検索可能 + backfill** | 移行時に完全なメタデータを待たない。フォルダ構造由来の `category` + ファイル名由来の `title` で一括取り込み → **即検索可能**に。`keywords`/`summary` は LLM が裏で順次 backfill。段階的開示は最小メタデータでも動作し、取りこぼしはベクトル検索が補完 |
| **一括取り込みジョブ化** | `scripts/ingest.mjs` を「単発」→「ディレクトリ再帰 + ワーカープール」に拡張。変換は CPU 並列、LLM 呼び出しは RPM 制限内で並列度を絞る。content-hash チェックポイントで**再開可能**に（途中失敗しても続きから） |
| **index 書き込みの O(n) 化** | `upsertIndex` の「1 件ごとに index.json 全体を書き換え（O(n²)）」を「ジョブ末尾で一括書き込み（O(n)）」に |
| **LLM enrichment の高速化** | Gemini Batch API（非同期・低単価・高スループット）でフロントマター生成をまとめて投入。embedding 生成（現状は別工程の `scripts/generate-embeddings.mjs`）も取り込みジョブに統合 |
| **増分取り込み + 重複排除** | content-hash で変更ファイルのみ再取り込み、完全重複は除外。初回移行のみ一度きりのコスト、運用フェーズは差分のみ → 文書が増え続けても取り込みが重くならない |

### 12.3 障壁2の解決方針：検索レイテンシをコーパスサイズから切り離す

per-query コストを文書件数に**非依存**にすることを原則とする。

| 方針 | 内容 | 設計整合 |
|---|---|---|
| **Step 1 前段に pre-filter 層** | LLM に渡す前に決定的な recall 層（メタデータ転置インデックス ∪ ベクトル ANN ∪ BM25 の和集合）で候補を 50〜100 件に固定。Step 1 のプロンプトが件数非依存になる | 段階的開示の前に 1 段足すだけ。Step 1〜3 のロジックは流用。メタデータ駆動を主（§2）に維持 |
| **ベクトル検索の ANN 化** | 全件線形コサインを HNSW（pgvector / hnswlib）に置換し sublinear 検索に | §6「500 件超で pgvector 移行」の具体化 |
| **index.json のシャーディング** | 単一ファイルをカテゴリ別 `documents/index/<category>.json` に分割。検索時は必要シャードのみロード、PR コンフリクトのホットスポットも分散 | `loadIndex()` のインターフェースは保持し内部実装のみ差替え |
| **インデックスのウォーム化** | ANN / メタインデックスを起動時 1 回ロードして常駐。大ファイルの毎リクエスト `JSON.parse`（現 `loadEmbeddings`）を廃止 | サーバレスのコールドスタート対策 |
| **クエリ結果キャッシュ** | FAQ 型の繰り返しクエリ向けに `query→候補` / `query→回答` を短 TTL でキャッシュ。人気クエリは LLM 呼び出しゼロで即応 | プロンプトキャッシュ（§4-C）と併用 |

段階的開示により Step 3（回答生成）は選択セクション本文のみを読むため、回答生成のコスト・速度は**もとから件数非依存**。pre-filter + ANN で Step 1 も件数非依存にすれば、パイプライン全体が「5,000 件でも 50,000 件でも検索速度が変わらない」構造になる。

### 12.4 アーキテクチャへの影響

§3 のハイブリッド検索（②）と段階的開示（③）の間に「pre-filter（粗い候補絞り込み）」が 1 段挿入される:

```
② ハイブリッド検索
   ↓ pre-filter: メタデータ ∪ ベクトル ANN ∪ BM25 で候補を 50〜100 件に固定
③ 段階的開示
   Step 1: 候補抽出（LLM が pre-filter 済みプール内で TOP-3 確定）
   Step 2: セクション本文取得
   Step 3: 回答生成
```

### 12.5 段階導入順

| 順 | 項目 | 理由 |
|---|---|---|
| 1 | 一括取り込みジョブ（並列・再開可能） | 障壁1の中核。他施策から独立して着手でき、効果が即見える |
| 2 | Step 1 前段の pre-filter 層 | 障壁2の中核。これ無しでは他のスケール施策が活きない |
| 3 | index.json シャーディング | pre-filter / ANN の前提を整える |
| 4 | ベクトルストアの ANN 化（pgvector / hnswlib） | 件数が実際に増えた時点で（§6, §10 整合） |
| 5 | ウォーム化・クエリキャッシュ・Batch API enrichment | 仕上げの最適化 |

各項目は **単独で価値が出る単位**に切る（§5 の原則を踏襲）。途中で止めても、その時点のコーパス規模では改善された状態を維持する。

## 13. 基幹システム連携への展望（マスタ値との整合）

§1〜§12 は本プロジェクト単体（Markdown コーパス + GitHub バックエンド）での完結を前提にしている。一方、導入先の企業に**既に基幹システム（ERP・人事給与システム等）が稼働している**ケースでは、家賃補助・各種手当・上限額といった「値」の正本は基幹システムのマスタにある（system of record）。社内文書（規程・マニュアル）にも同じ値が散在しており、現行コーパスでも 1 つの金額が複数文書に重複している（例: `30,000円` は慶弔見舞金規程・経費精算マニュアル・給与規程など 6 文書に出現）。基幹側でマスタが更新されても文書側は手動更新が必要なため、両者の**乖離（ドリフト）**が避けられない。

本節は、この乖離を本プロジェクトの売り（**チャットベースで簡単に検索・編集できる**こと）を壊さずに埋めるための連携方針をまとめる。**本節は案・懸念・展望であり、現時点では未着手・未決定**。実装するとしても §5 のフェーズ群とは独立した将来トラックとして扱う。

### 13.1 基本方針 — 連携は「修正案の入口」に限定する

連携を「データ結合」ではなく「**修正案の入口**」に限定する。役割を次のように分ける。

| 担い手 | 役割 |
|---|---|
| 基幹システム | 「値の真実」（金額・上限額などの正本）|
| 本プロジェクト | 「文章の真実」（規程・マニュアルの文面）を保ち、人が直すのを支援 |

- 連携は**一方向（基幹 → 文書）・読み取りのみ**。文書側から基幹へは書き戻さない。
- 連携の出力は「マスタ変更 → 文書の**修正案 PR の下書き**」までに留める。反映は既存の人間レビュー（§4-D / §4-F / §10）を必ず経る。自動マージはしない。
- 本プロジェクト側に DB は持たず、基幹システムの値を API 等で**読むだけ**。文書本体は GitHub のまま、監査も commit log のまま。これにより §10「DB で文書本体を管理しない」と整合する。

### 13.2 想定フロー

```
基幹システムでマスタレコード変更（例: 家賃補助上限 50,000 → 55,000）
  ↓ トリガー（webhook イベント / 定期 diff ポーリング）
  ↓ 旧値を含む文書・箇所を特定（find_text_occurrences 相当）
  ↓ propose_related_edit で {find, replace, reason}[] の一括 PR を下書き
      reason に基幹側の変更根拠を明記（例:「基幹システムで家賃補助上限が改定されたため」）
  ↓ チャット / 編集 UI（Monaco DiffEditor）で人間がレビュー・調整・承認
  → merge で反映
```

最後の「チャットで簡単にレビュー・編集」こそが本プロジェクトの売りであり、連携は**その手前にネタを供給するだけ**でレビュー体験は一切変えない。受け身の PR だけでなく、エージェントがチャットで能動的に「基幹システムで◯◯が変わりました。N 文書に旧値が残っています。修正案 PR を作成しました」と通知する見せ方も考えられる。

### 13.3 懸念点（未解決の検討事項）

| 懸念 | 内容 |
|---|---|
| **マッピング問題** | 基幹マスタの項目 → どの文書のどの箇所、の対応付けが必要。同じ数字でも文書により意味が異なり（`30,000円` が「経費の事前申請しきい値」だったり「住宅取得祝金」だったり）、素朴な全文一致は誤爆する。基幹項目 ↔ 文書箇所を結ぶ**軽量なマッピング定義**が要る。これは DB ではなく git 管理のメタデータ（§10 整合）|
| **トリガー手段** | 基幹システムが変更イベント（webhook 等）を出せるかは導入先依存。出せない場合は定期的に値を取得して差分検知（ポーリング）。鮮度と実装コストが変わる |
| **値だけでない変更** | 改定が金額だけなら find/replace で足りるが、支給条件など文章構造に及ぶ変更は単純置換にならない。提案品質のばらつきは人間レビュー段階で吸収する |
| **接続性・セキュリティ** | 基幹システムは社内ネットワーク内に閉じていることが多く、Cloudflare Workers から直接到達できない可能性がある。中継（社内エージェント / VPN / バッチエクスポート）と、基幹データ読み取りの認証情報管理が課題 |
| **ライブ参照は採らない** | 文書を基幹 DB のプレースホルダ（テンプレート）にして実行時に値を差し込む方式は採用しない。検索が返すセクション本文の自己完結性・セクション単位の精密な引用・git による監査が崩れ、可用性が基幹システムに依存するため。段階的開示（§2 / §3）の構造を壊す |

### 13.4 アーキテクチャへの影響

§3 ①（取り込みパイプライン）と並ぶ新しい入力経路として「**外部システム連携コネクタ**」が加わるだけで、編集の中核フローは既存のものを再利用する。

- 文書の特定・編集 PR 化は既存の `find_text_occurrences` / `propose_related_edit`（§4-F）をそのまま使う。
- 新規実装は「トリガー（変更検知）」と「マッピング（基幹項目 ↔ 文書箇所）」の 2 点のみ。
- 疎結合を保つ: コネクタは `lib/` の既存編集モジュールを import するだけにとどめ、既存パイプラインと共存させる（§4-F の MCP コネクタと同じ方針）。

### 13.5 段階導入順（将来）

| 順 | 項目 | 理由 |
|---|---|---|
| 1 | 手動トリガーの PoC | 基幹の新旧値を手入力 → `find_text_occurrences` で該当文書を検出 → 修正案 PR。マッピング定義なしで連携体験を検証できる |
| 2 | マッピング定義の導入 | 基幹項目 ↔ 文書箇所の git 管理レジストリ。「同じ数字・違う意味」の誤爆を防ぐ中核 |
| 3 | 自動トリガー | webhook（基幹が対応する場合）or 定期ポーリングで差分検知 → 修正案 PR を自動下書き |
| 4 | チャットへの能動通知 | エージェントがチャットで乖離と修正案 PR を能動的に知らせる |

各項目は **単独で価値が出る単位**に切る（§5 の原則を踏襲）。1 で止めても「基幹の値を入れれば該当文書がわかる」状態が手に入り、現状より改善される。

## 14. Colletie（テクノブレイブ）連携への展望（外部ナレッジソースの取り込み）

§13 は「基幹システム（値の正本）」との連携だった。本節は別種の外部システム——**他社のナレッジ管理 SaaS に既に蓄積された社内ナレッジ**——を本プロジェクトのコーパスに取り込む連携をまとめる。具体例としてテクノブレイブ株式会社の **Colletie（コレッティー）** を取り上げる。**本節は案・展望であり、現時点では未着手・未決定**。実装するとしても §5 のフェーズ群とは独立した将来トラックとして扱う。

### 14.1 Colletie とは

Colletie はテクノブレイブ株式会社のクラウド型ナレッジマネジメント SaaS（2023 年 4 月提供開始）。

| 観点 | 内容 |
|---|---|
| 投稿形式 | Markdown 記法（見出し・太字・コードブロック・引用・箇条書き）|
| 整理単位 | ルーム（room）+ タグ。FAQ・WEB ドキュメント・ノート |
| 検索 | キーワード / 期間 / ユーザー / 自然言語検索 |
| その他機能 | コメント、テンプレート、タスク・プロジェクト管理、目標管理（MBO）、データ分析、通知 |
| 提供形態 | クラウド SaaS、30 日無料トライアル |

本プロジェクトと Colletie は機能的に重なる領域がある（§11 の InfoCraft 比較と同じ「市販 SaaS」枠）。ただし両者は**役割が異なる**と捉えるのが連携の出発点になる。

| 担い手 | 役割 |
|---|---|
| Colletie | 人がナレッジを**書き溜める場**（執筆・日常更新の UI が成熟）|
| 本プロジェクト | 段階的開示で**精密に検索・横断回答**し、構造化編集（`{find, replace, reason}`）を PR 化する**検索・編集エージェント層** |

→ **Colletie に既に社内ナレッジが蓄積済みの企業**に対し、本プロジェクトを「上に乗せる検索 AI 層」として提供できる。この見方では Colletie は競合ではなく**コンテンツソース**になる。

### 14.2 基本方針 — Colletie は取り込みパイプラインの新しい入力ソース

連携は §3① 取り込みパイプラインの**新しい入力ソース**として実装する。§13 と同じ原則を踏襲する。

- 連携は**一方向（Colletie → 本プロジェクト）・読み取りのみ**。本プロジェクトから Colletie へ書き戻さない（双方向同期は採らない。理由は 14.4）。
- Colletie は **Markdown ネイティブ**なので、§4-B「多形式 → MD 変換」のうち変換コストがほぼゼロ。既存の `convertMarkdown`（`lib/ingest-core.ts` / `scripts/ingest.mjs`）がそのまま使える。
- フロントマター生成も全部を LLM に頼らない。Colletie のメタデータを決定的にマッピングできる:

| Colletie | → 本プロジェクトのフロントマター |
|---|---|
| ルーム名 | `category`（マッピング定義経由で決定的に。LLM 不要）|
| タグ | `keywords` の初期値 |
| 投稿タイトル | `title` |
| 投稿本文 | body（H2 → セクションマーカー注入は既存の `injectSectionMarkers`）|
| 投稿日時 | `created_date` / `last_updated` |

→ LLM enrichment（`generateFrontmatterWithLlm`）は `summary` と `keywords` 補完だけに絞れる。§12.2「Day 1 検索可能 + backfill」と同じく、最小メタデータで即取り込み → `summary` は裏で backfill も可能。

- 取り込んだ文書は既存の `ingest_documents` MCP ツール（§4-F）/ `scripts/ingest.mjs` 経由で **PR 化**。反映は人間レビューを経る（自動マージなし、§10 整合）。

### 14.3 想定フロー

```
Colletie のルーム / 投稿
  ↓ 取得（公開 API があれば API、なければ手動エクスポート）
  ↓ ルーム→category・タグ→keywords をマッピング（決定的）
  ↓ Markdown 本文に H2 セクションマーカー注入（既存 injectSectionMarkers）
  ↓ 不足メタデータ（summary 等）を LLM enrichment
  ↓ content-hash で増分判定（既出・無変更はスキップ）
  ↓ ingest_documents で新規 / 更新文書をまとめて PR 下書き
  → チャット / 編集 UI で人間レビュー → merge で反映
```

### 14.4 懸念点（未解決の検討事項）

| 懸念 | 内容 |
|---|---|
| **公開 API の有無（最大の依存）** | Colletie の開発者向け公開 API は現時点の公開情報では確認できない。**テクノブレイブへの確認が連携可否の前提**。API があれば自動同期、無ければ手動エクスポート（Markdown / CSV）→ 既存 `scripts/ingest.mjs` というフォールバックで PoC は成立する |
| **ルーム ↔ category マッピング** | Colletie のルーム構成と本プロジェクトのカテゴリ体系は一致しない。素朴な自動割当は誤分類を生む。ルーム ↔ category の**軽量マッピング定義**（git 管理のメタデータ、§10 整合）が要る |
| **取り込み対象の線引き** | Colletie のコメント・タスク・MBO・データ分析は取り込み対象外を推奨。スレッド型コメントやタスクは段階的開示の「文書 → セクション」構造に合わない。取り込むのはナレッジ / WEB ドキュメント本文のみ |
| **増分取り込み** | 全件再取り込みは PR を肥大化させる。content-hash で変更投稿のみ再取り込み（§12.2 の増分取り込みと整合）。Colletie 側の更新日時が取れれば併用 |
| **認証情報管理** | Colletie の API キー / 認証情報を環境変数化（§2「LLM 抽象化を最小限残す」と同じ方針）。Cloudflare Workers から Colletie に到達できるかは要確認（§13.3 の接続性懸念と同種）|
| **双方向同期は採らない** | 本プロジェクトから Colletie へ書き戻す方式は採用しない。正本がどちらか曖昧になり、§13.3「ライブ参照は採らない」と同じく可用性・監査（git commit log）の一貫性が崩れる。導入先は「Colletie を執筆の正本に保つ」か「本プロジェクト（GitHub）へ正本を移す」かを**一度だけ**決め、移行後は一方向に固定する |

### 14.5 アーキテクチャへの影響

§3① 取り込みパイプラインと並ぶ新しい入力経路として「**Colletie 取り込みコネクタ**」が 1 つ加わるだけ。中核フロー（変換 → セクション注入 → フロントマター → index 更新 → PR）は既存のものを再利用する。

- 変換・セクション注入・PR 化は既存の `lib/ingest-core.ts` / `ingest_documents`（§4-F）をそのまま使う。
- 新規実装は「コネクタ（Colletie API / エクスポート読み取り）」と「マッピング（ルーム ↔ category）」の 2 点のみ。
- 疎結合を保つ: コネクタは `lib/` の既存取り込みモジュールを import するだけにとどめる（§4-F の MCP コネクタ、§13.4 と同じ方針）。

### 14.6 段階導入順（将来）

| 順 | 項目 | 理由 |
|---|---|---|
| 1 | 手動エクスポート → ingest の PoC | Colletie からエクスポートした Markdown を既存 `scripts/ingest.mjs` に通すだけ。**API 不要**で連携体験と検索品質を検証できる |
| 2 | ルーム ↔ category マッピング定義 | git 管理のレジストリ。誤分類を防ぐ中核 |
| 3 | API 経由の自動取り込み | Colletie に公開 API があれば、webhook / 定期ポーリングで差分を取得し PR を自動下書き |
| 4 | 増分取り込み + 重複排除 | content-hash で変更投稿のみ再取り込み。運用フェーズは差分のみ |

各項目は **単独で価値が出る単位**に切る（§5 の原則を踏襲）。1 で止めても「Colletie のナレッジを本プロジェクトの段階的開示検索で引ける」状態が手に入り、現状より改善される。
