## 1. 権限付与原則
<!-- section_id: sec_1 -->

### 第1条(目的)
本規程は、株式会社サンプル(以下「当社」という。)の情報システム及び情報資産に対するアクセス権限の付与、変更、剥奪、棚卸等に関する基準と手続を定め、不正アクセス、内部不正、過剰な権限による情報漏えい等のリスクを低減することを目的とする。本規程はJIS Q 27001:2023 附属書A 5.15〜5.18、8.2〜8.5、JIS Q 27002:2024該当箇条、NIST SP 800-53 Rev.5 AC群、不正アクセス行為の禁止等に関する法律に準拠する。

### 第2条(基本原則)
アクセス権限の付与は、以下の原則を満たすこと。
1. **最小権限の原則(Principle of Least Privilege)**:利用者には業務遂行に必要な最小限の権限のみを付与する。「念のため」「将来必要になる可能性がある」を理由とした過剰権限付与を禁止する。
2. **知る必要性の原則(Need-to-Know)**:情報へのアクセスは、業務上知る必要がある者に限定する。組織図上の上位者であっても、業務関連性のない情報へのアクセス権限は付与しない。
3. **職務分掌(Segregation of Duties, SoD)**:相互牽制が必要な業務(申請者と承認者、開発者と本番リリース実施者、出納担当と経理記帳担当、購買発注者と検収者等)について、同一人物に両権限を付与しない。
4. **デフォルト拒否(Default Deny)**:明示的に許可されていない権限は付与しない。
5. **ゼロトラストの原則**:社内ネットワーク内からのアクセスであっても認証・認可を都度実施する。
6. **共用ID原則禁止**:利用者は自己専用のIDを使用すること。サービスアカウントを除き共用IDの作成を禁止する。

### 第3条(役割と責任)
| 役割 | 責任 |
|------|------|
| CISO | アクセス制御方針の最終承認、重大な権限付与(基幹システム特権)の最終承認 |
| ISMS事務局 | 規程整備、教育、棚卸結果集約、是正監督 |
| システムオーナ | 所管システムのアクセス制御ポリシー定義、ロール設計 |
| システム管理者 | アクセス権限付与・変更・剥奪の技術的実施、ログ管理 |
| 申請者の直属長 | 申請内容の妥当性審査、業務必要性の確認、SoD抵触の確認 |
| 利用者 | 自IDの厳重管理、パスワード非共有、不正利用の即時報告 |
| 内部監査室 | アクセス権限管理状況の独立評価、年次監査 |

### 第4条(認証要件)
全社員ID(Active Directory / Entra ID連携)については、以下の認証要件を必須とする。
- パスワード長:14文字以上、辞書攻撃に強い複雑性要件(NIST SP 800-63B準拠)
- 多要素認証(MFA):全システム必須(認証アプリ又はFIDO2セキュリティキー)
- パスワード使い回し検出(漏えいパスワードDB照合)
- アカウントロック:認証失敗5回で30分ロック
- セッションタイムアウト:無操作30分(特権セッションは15分)
- シングルサインオン(SSO):業務システムは原則SSO連携必須

## 2. 権限申請フロー
<!-- section_id: sec_2 -->

### 第5条(標準申請フロー)
アクセス権限の新規付与・変更・解除は、原則として社内ワークフロー(ServiceNow又は同等ツール)を通じて以下の手順で行う。

```
[申請者本人] 
   ↓ 申請(対象システム、必要権限、業務目的、利用期間を記載)
[直属長(課長以上)] 
   ↓ 承認(業務必要性、最小権限性、SoD抵触有無を確認)
[システムオーナ] 
   ↓ 承認(対象システムのアクセス制御ポリシーとの整合確認)
[システム管理者] 
   ↓ 付与実施(技術的設定、付与記録、ログ取得)
[ISMS事務局(重要システムのみ)] 
   ↓ 事後モニタリング(月次レビュー)
[申請者へ完了通知]
```

申請には少なくとも以下を記載する。
- 申請者氏名、所属、社員番号
- 対象システム名、必要となる権限ロール(参照、登録、変更、削除、管理者等)
- 業務上の必要性、対象データ範囲
- 利用開始予定日、利用終了予定日(プロジェクト終了等期限がある場合)
- 既に保有する他システム権限との関係(SoD自己申告)

### 第6条(重要システムの追加承認)
以下のシステム・データに対する権限付与は、上記フローに加えてCISOの追加承認を必要とする。
- 経理・財務基幹システム(会計仕訳、決算、振込実行権限)
- 人事システム(給与、人事評価、マイナンバー)
- 顧客個人情報データベース(顧客マスタ、購入履歴)
- 開発本番環境(リリース、データベース直接アクセス)
- セキュリティ機器・SIEM・EDR管理コンソール
- クラウド管理コンソールの管理者ロール
- 暗号鍵管理システム(HSM、KMS)
- ドメイン管理者、グローバル管理者ロール(Microsoft 365、AWS root等)

### 第7条(緊急時の権限付与)
障害対応・インシデント対応等で平常時の承認フローを待てない場合、CSIRT責任者又は当直管理者の口頭承認により権限付与を行い、事後72時間以内に正規のワークフローで追認する。緊急時付与のログは月次でISMS事務局がレビューする。

### 第8条(委託先・派遣社員の権限管理)
委託先・派遣社員に対する権限付与も、当社プロパー社員と同様のワークフローを適用する。ただし以下の追加要件を課す。
- 契約書又は派遣個別契約書に記載された業務範囲内であること
- 契約期間に連動した自動失効を設定すること
- 委託先所属企業の在籍確認を年1回実施すること
- アクセスログを通常社員より厳重に保管(7年間)

## 3. 異動・退職時
<!-- section_id: sec_3 -->

### 第9条(異動時の権限変更)
役職員の異動が発令された場合、人事部は異動発令と同時に当該役職員のリストをISMS事務局及び情報システム部へ連携する。システム管理者は以下のSLAで対応する。
1. 異動元部署固有の権限:**異動日から72時間以内**に剥奪
2. 異動先部署で必要な権限:異動日までに直属長から申請、異動日に有効化
3. 兼務発令の場合:両部署の権限保持の妥当性をSoD観点で確認の上、保持可否を判断

### 第10条(退職・契約終了時)
退職、契約満了、解雇等により当社との雇用・契約関係が終了する場合、以下を実施する。
1. **無効化SLA**:退職日の業務終了時点(当日23:59)までに、当該役職員の全社員ID、業務システムID、VPN、SaaSアカウント、メール、チャットツール、各種クラウド権限を**一括無効化**する。
2. **物理アセット回収**:社員証、PC、スマートフォン、USBトークン、入館カード、書類等を退職日までに回収する。
3. **再雇用・出戻りの場合**:新規入社と同等の手続を行い、旧IDを再有効化することなく新規発行する。
4. **退職予告期間中の権限制限**:競合転職、リスクのある退職等の場合、上長判断とCISO承認により、退職予告日から退職日までの間に重要システムへのアクセスを制限することができる。
5. **退職後のメール転送禁止**:退職者個人へのメール自動転送は原則禁止。業務継続のため、上長指定者への転送設定を最大3か月間に限り認める。

### 第11条(休職時)
3か月を超える休職に入る役職員のIDは、休職開始から1週間以内に無効化又はサスペンド状態とする。復職時に再有効化する。

### 第12条(死亡退職)
役職員の死亡退職の場合、人事部の連絡を受け、即時にID無効化を行う。当該者宛のメール、保管データの取扱いは、業務継続上必要な場合のみ上長判断で開示し、私的内容には立ち入らない。

### 第13条(棚卸との照合)
人事部は四半期に1回、現役職員リストをISMS事務局に提供し、Active Directory及び主要SaaS上のアカウントとの突合監査を実施する。差分(退職者の未削除アカウント、不明アカウント等)は2週間以内に解消する。

## 4. 特権ID管理
<!-- section_id: sec_4 -->

### 第14条(特権IDの定義)
本規程における「特権ID」とは、以下のいずれかに該当する権限を有するIDをいう。
- OSのroot、Administrator、ローカル管理者
- データベースのDBA、sa、sysdba
- ネットワーク機器のenable、admin
- クラウドサービスのrootユーザ、グローバル管理者、特権ロール
- アプリケーション管理者、ユーザ一括管理権限
- セキュリティ製品の管理コンソール権限
- 暗号鍵・証明書管理権限

### 第15条(原則と管理方式)
1. **個人帰属の原則**:特権権限も原則として個人IDに付与し、共用特権IDの使用を禁止する。やむを得ず共用となる場合(製品仕様上の制約等)は、特権ID管理ツール(CyberArk等のPAM製品)を経由した払い出し方式を採用する。
2. **ジャストインタイム(JIT)アクセス**:特権権限は常時付与せず、必要な期間のみ申請に基づき付与する。デフォルト権限保有時間は最大4時間とし、超過時は再申請を要する。
3. **特権アクセスワークステーション(PAW)**:基幹システム特権操作は、業務PCではなく専用の隔離端末から実施する。
4. **多要素認証強制**:特権IDログイン時はFIDO2セキュリティキーによる認証を必須とする。
5. **操作録画**:特権セッションは画面録画及びコマンドログを全件取得する。録画データは改ざん防止ストレージに保管する。
6. **パスワード自動ローテーション**:共用特権IDのパスワードは使用都度自動変更する。
7. **特権IDの保有者見直し**:四半期に1回、特権ID保有者一覧をシステムオーナ及びCISOがレビューし、不要権限を剥奪する。

### 第16条(緊急アカウント:Break Glass)
クラウドのrootユーザ、ドメインのEnterprise Adminなど、平常時には使用しない最高権限アカウントについては、以下を整備する。
- パスワード/MFAリカバリコードを物理金庫に分割保管(2名以上の立会で開錠)
- 使用は重大障害時に限定し、使用記録を残す
- 使用後は速やかにパスワードリセット
- 月次で存在確認と認証情報の動作確認

### 第17条(ログ管理)
特権ID関連ログは以下を遵守する。
- ログイン/ログアウト、権限変更、重要操作の全てを記録
- ログの完全性保護(改ざん検知、別系統SIEMへの転送)
- 保管期間:**最低3年間**(個人情報を取り扱うシステムは7年間)
- 月次でCSIRT及び内部監査室がレビュー、異常検知時はインシデント手順を発動

## 5. 定期棚卸
<!-- section_id: sec_5 -->

### 第18条(棚卸頻度と対象)
ISMS事務局は、全社のアクセス権限について以下の頻度で棚卸を実施する。

| 棚卸対象 | 頻度 | 実施時期 | 実施主体 |
|----------|------|---------|---------|
| 一般ユーザ権限(全社員ID) | 半期に1回 | 6月、12月 | 各部門長 |
| 重要システム権限(第6条対象) | 四半期に1回 | 3月、6月、9月、12月 | システムオーナ+CISO |
| 特権ID | 四半期に1回 | 3月、6月、9月、12月 | システムオーナ+CISO |
| 委託先・派遣社員ID | 四半期に1回 | 3月、6月、9月、12月 | 契約管理部門 |
| サービスアカウント | 年1回 | 9月 | システム管理者 |
| クラウドリソースのIAM | 半期に1回 | 6月、12月 | クラウド管理者 |

### 第19条(棚卸手順)
1. ISMS事務局は、対象期間開始の2週間前に棚卸対象リスト及び実施要領を関係部門に配布する。
2. 各部門責任者は、配布されたID・権限一覧を確認し、各利用者の業務必要性を判定する。
3. 不要権限、過剰権限、SoD抵触権限を発見した場合、棚卸シートに「要剥奪」「要変更」と記録し、システム管理者へ剥奪・変更依頼を起票する。
4. 部門責任者は棚卸完了後、棚卸結果を電子署名付きでISMS事務局に提出する(責任者証跡として保管)。
5. ISMS事務局は全社集計し、ISMS委員会及び内部監査室に報告する。

### 第20条(是正期限)
棚卸により発見された不要権限・過剰権限は、原則として棚卸完了日から**30日以内**に是正する。是正困難なものについてはリスク受容の妥当性をCISOが個別判断する。

### 第21条(棚卸記録の保管)
棚卸記録(棚卸シート、是正依頼、是正完了記録)は、外部監査・ISMS審査の証跡として5年間保管する。

### 第22条(継続的モニタリング)
棚卸とは別に、SIEM及びID Governanceツールにより以下の継続的モニタリングを実施する。
- 90日以上未ログインのアカウント検出(自動無効化候補)
- 退職予定者リストとの突合
- 異常な権限付与(短期間に複数権限取得、深夜の権限変更等)の検知
- SoD抵触の自動検出

### 第23条(違反時の措置)
本規程に違反した役職員には、就業規則及び懲戒規程に基づく処分を行うことがある。特に、他人のIDによる成りすまし、共有禁止IDの共有、特権の私的利用は重大な違反として厳正に処分する。委託先の違反は契約に基づく契約解除及び損害賠償の対象となる。
