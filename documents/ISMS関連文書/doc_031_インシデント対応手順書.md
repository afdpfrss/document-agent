## 1. インシデント分類
<!-- section_id: sec_1 -->

### 第1条(目的及び根拠)
本手順書は、株式会社サンプル(以下「当社」という。)の「情報セキュリティ基本方針」(doc_029)に基づき、情報セキュリティインシデント(以下「インシデント」という。)の検知、報告、初動対応、封じ込め、根絶、復旧、教訓化に関する手順を定めることを目的とする。本手順書はJIS Q 27001:2023 附属書A 5.24〜5.28、JIS Q 27035シリーズ、NIST SP 800-61 Rev.2「Computer Security Incident Handling Guide」、内閣サイバーセキュリティセンター(NISC)「政府機関等のサイバーセキュリティ対策のための統一基準群」を参考としている。

### 第2条(用語定義)
- **イベント**:情報システム、サービス又はネットワークにおける、識別された発生事象。
- **インシデント**:単独又は連続して発生する、事業運営や情報セキュリティを脅かす可能性が高い、望ましくないイベント。
- **重大インシデント**:後述のレベルL1又はL2に該当するもの。
- **CSIRT(Computer Security Incident Response Team)**:インシデント対応専門チーム。当社では情報システム部セキュリティ課に常設。
- **PSIRT(Product Security Incident Response Team)**:自社製品の脆弱性対応チーム。
- **IoC(Indicator of Compromise)**:侵害指標。

### 第3条(インシデント分類表)
インシデントは、以下のカテゴリ及び影響度レベルにより分類する。CSIRT責任者が初動段階で仮判定し、調査進展に応じ随時見直す。

#### カテゴリ
| カテゴリ | 例 |
|----------|----|
| マルウェア感染 | ランサムウェア、Emotet、トロイの木馬、ワーム |
| 不正アクセス | 認証情報窃取、ブルートフォース、権限昇格、内部不正 |
| 情報漏えい | メール誤送信、SNS誤投稿、USB紛失、クラウド誤設定公開 |
| サービス停止 | DDoS攻撃、ハードウェア障害、クラウド障害、ネットワーク断 |
| 物理的事象 | PC・スマートフォン盗難、書類紛失、不正侵入、自然災害 |
| ソーシャルエンジニアリング | フィッシング、ビジネスメール詐欺(BEC)、ヴィッシング |
| 脆弱性 | 自社製品・利用OSSの脆弱性公表、ゼロデイ |
| サプライチェーン | 委託先からの侵害、ソフトウェア供給網侵害 |

#### 影響度レベル及び対応SLA
| レベル | 名称 | 定義 | 1次報告SLA | 経営報告SLA | 主たる指揮 |
|--------|------|------|-----------|-------------|------------|
| L1 | 緊急(Critical) | 個人情報1,000件以上漏えい又は基幹システム全停止、社会的影響大、報道リスク高、要配慮個人情報を含む | 検知から15分以内 | 1時間以内 | CISO直轄、対策本部設置 |
| L2 | 重大(High) | 個人情報1,000件未満漏えい、複数システムへの影響、主要顧客への影響 | 30分以内 | 3時間以内 | CSIRT責任者+CISO |
| L3 | 重要(Medium) | 限定的な影響(単一部署、単一システム)、機微情報含まず | 2時間以内 | 翌営業日 | CSIRT |
| L4 | 軽微(Low) | 個別端末感染、軽微な誤送信(社内のみ)等、迅速封じ込め可能 | 当日中 | 週次報告に集約 | 情報システム部 |

### 第4条(CSIRT体制)
当社CSIRTは以下のメンバーで構成する。重大インシデント発生時には、CISOを本部長とする「セキュリティ対策本部」を招集する。
- **CSIRT責任者**:情報システム部セキュリティ課長(指揮、対外連絡窓口=PoC)
- **技術リーダ**:セキュリティアーキテクト(フォレンジック、封じ込め技術判断)
- **インシデントハンドラ**:セキュリティ課員(24時間365日オンコール体制、3名×3シフト)
- **法務リエゾン**:法務部担当者(法令対応、当局報告判断、契約関連)
- **広報リエゾン**:コーポレートコミュニケーション部担当者(対外公表、報道対応)
- **人事リエゾン**:人事部担当者(内部不正案件、懲戒検討)
- **事業部リエゾン**:該当事業部の情報セキュリティ責任者(業務影響評価)
- **外部委託**:フォレンジック調査会社(年間契約、SLA:24時間以内現地到着)

## 2. 初動対応
<!-- section_id: sec_2 -->

### 第5条(検知経路)
インシデントの検知は、以下の経路により行われる。
1. EDR/XDR(全社配備)、SIEM、IDS/IPS、WAF、DLPからの自動アラート
2. SOC(24時間外部監視)からのエスカレーション
3. 役職員からの通報(社内ヘルプデスク、専用通報窓口:incident-report@sample.example、内線XXXX)
4. 取引先、顧客、外部研究者からの通報(脆弱性開示ポリシーに基づく窓口)
5. JPCERT/CC、IPA、各種ISACからの注意喚起
6. 標的型攻撃メール訓練の真正報告

### 第6条(発見者の初動義務)
インシデントを発見し又はその疑いを認識した役職員は、直ちに以下を実施しなければならない。
1. 自端末の場合、ネットワークから物理的に切断(LANケーブル抜線、Wi-Fi切断)する。ただし**電源は切らない**(メモリフォレンジック証跡保全のため)。
2. 直属上長へ口頭又はチャットで第一報する。
3. CSIRT通報窓口(専用フォーム又は電話)へ通報する。
4. 当該事象に関するメール、画面キャプチャ等を可能な範囲で保全する。
5. SNSへの投稿、関係者外への口外を厳禁とする。

### 第7条(初動チェックリスト)
CSIRTインシデントハンドラは、通報受領後直ちに以下のチェックリストに沿って対応する。

| 項目 | 内容 | 完了確認 |
|------|------|---------|
| 1 | 通報内容の記録(発生日時、発見者、事象、影響範囲の初期推定) | □ |
| 2 | チケット起票(CSIRT管理システム)、インシデント番号採番 | □ |
| 3 | 影響度レベル仮判定(L1〜L4) | □ |
| 4 | CSIRT責任者へエスカレーション | □ |
| 5 | 関係端末・サーバの隔離(EDR隔離機能、ネットワークACL変更) | □ |
| 6 | 証拠保全(ディスクイメージ取得、メモリダンプ、関連ログ抽出) | □ |
| 7 | アカウントの強制パスワード変更、トークン失効 | □ |
| 8 | 影響範囲の特定(ログ分析、IoC横展開調査) | □ |
| 9 | 関係者連絡(事業部リエゾン、法務、広報) | □ |
| 10 | タイムラインの記録開始(以後の全行動を分単位で記録) | □ |

### 第8条(証拠保全の原則)
フォレンジック証跡として法的有効性を確保するため、以下を遵守する。
- 原本に手を加えず、必ずビット単位コピーで作業する(write-blocker使用)。
- ハッシュ値(SHA-256)を取得し、保全前後で一致を確認する。
- 取扱者、受領日時、保管場所のChain of Custody(証跡連鎖)を記録する。
- 保全データは専用金庫又は暗号化ストレージで最低7年間保管する。

## 3. 報告フロー
<!-- section_id: sec_3 -->

### 第9条(社内報告ライン)
インシデント情報は、以下の経路で報告する。レベルに応じ並列で同時報告を行う。

```
[発見者] → [直属上長] → [CSIRT(セキュリティ課)] → [CSIRT責任者]
                                                            ↓
                                                          [CISO]
                                                            ↓
                                            [経営層(代表取締役・関係取締役)]
                                                            ↓
                                                  [取締役会・監査役会報告]
```

L1案件は、CSIRT責任者からCISOへ15分以内、CISOから代表取締役へ1時間以内に直接報告する。

### 第10条(社外報告基準)
| 報告先 | 報告要否の判断基準 | 報告期限 | 根拠法令・契約 |
|--------|--------------------|----------|----------------|
| 個人情報保護委員会 | 個人データ漏えい等(要配慮、財産的被害、不正アクセス、1,000人超のいずれか該当時必須) | 速報:認識から3〜5日以内、確報:30日以内(不正アクセス案件は60日以内) | 個人情報保護法第26条、施行規則第7〜8条 |
| 本人(被害者) | 個人データ漏えい等の対象本人 | 状況把握後速やかに | 個人情報保護法第26条第2項 |
| 欧州データ保護当局(該当監督機関) | EU居住者の個人データ侵害 | 認識から72時間以内 | GDPR第33条 |
| 所轄警察署サイバー犯罪対策課 | サイバー攻撃、不正アクセス、内部不正等の刑事事案 | 速やかに | 不正アクセス禁止法、刑法 |
| JPCERT/CC | 攻撃手法の情報共有が有益な事案 | 任意、速やかに | ─ |
| IPA | 標的型攻撃、新種脆弱性の届出 | 任意、速やかに | ─ |
| 金融庁・経済産業省等 | 業法上の報告義務がある場合 | 各業法所定 | 各業法 |
| 取引先・顧客 | 顧客データに影響する場合、契約上の通知義務がある場合 | 契約所定(通常24〜72時間) | NDA、DPA、SLA |
| 報道機関(プレスリリース) | 社会的影響が大きく、自主開示が望ましい場合 | 広報・CISO・法務協議の上、適切なタイミング | ─ |

### 第11条(対外公表)
プレスリリース、お詫び文の発信は、コーポレートコミュニケーション部が起案し、法務部、CISO、代表取締役の承認を得て公表する。事実確認が不十分な段階での憶測による公表は避け、誠実かつ正確な情報開示を行う。

### 第12条(対応記録)
全てのインシデントについて、以下を記載した「インシデント対応報告書」を作成し、CSIRT管理システムに保管する。
- インシデント番号、検知日時、発見者、検知経路
- 事象の概要、原因、影響範囲(件数、対象)
- 時系列対応記録(タイムライン)
- 関係者一覧、外部報告履歴
- 復旧措置、再発防止策、責任部署、完了期限
- 学んだ教訓(Lessons Learned)

## 4. 復旧と再発防止
<!-- section_id: sec_4 -->

### 第13条(封じ込め・根絶・復旧の段階)
NIST SP 800-61 Rev.2のフェーズに準拠し、以下の段階を踏む。
1. **封じ込め(Containment)**:被害拡大を止める短期措置(隔離、通信遮断、アカウントロック)。短期封じ込め後、長期封じ込め(代替環境構築、フォレンジック完了までの維持)へ移行。
2. **根絶(Eradication)**:マルウェア駆除、不正アカウント削除、脆弱性パッチ適用、改ざんファイル復元。
3. **復旧(Recovery)**:クリーン環境での再構築、データのバックアップからの復元、段階的なサービス再開、再侵入監視の強化。
4. **事後対応(Post-Incident Activity)**:報告書作成、レビュー会議、再発防止策、教訓の組織共有。

### 第14条(根本原因分析:RCA)
重大インシデントについては、CSIRT責任者主導で根本原因分析(Root Cause Analysis)を実施する。手法として「なぜなぜ分析(5 Whys)」「特性要因図(フィッシュボーン)」を用い、技術的要因、運用上の要因、組織的要因、人的要因の各層で原因を特定する。

### 第15条(是正処置と水平展開)
RCAの結果に基づき、是正処置計画を策定する。計画には、対策内容、責任部署、期限、有効性測定方法を含める。同種事象の発生が他システム・他部署でも考えられる場合、ISMS事務局が水平展開を主導し、全社的な対策を講じる。

### 第16条(レビュー会議)
重大インシデント(L1・L2)の終結後、原則として2週間以内に「ポストモーテムレビュー(Blameless Post-mortem)」を開催する。本会議は、個人の責任追及ではなく、組織としての学びを得ることを目的とする。議事録は社内ポータルに公開し、教訓を全社で共有する。

### 第17条(再発防止策の有効性確認)
是正処置完了後、CSIRTは3か月後、6か月後、12か月後の3回、対策の有効性を確認する。同種事象が再発した場合は、対策不十分と評価し、追加対策を立案する。

### 第18条(訓練)
当社は以下の訓練を計画的に実施し、対応能力を維持向上する。
- 標的型攻撃メール訓練:全社員対象、年4回
- ランサムウェア対応机上演習:CSIRT及び経営層、年2回
- 危機広報訓練:広報・法務・CISO、年1回
- データセンター物理障害復旧訓練:インフラ部門、年1回
- 個人情報漏えい対応訓練(個情委報告含む):CSIRT・法務、年1回

### 第19条(規程・手順の見直し)
インシデントから得られた教訓は、本手順書を含む関連ISMS文書の改定に反映する。改定はISMS事務局が起案し、ISMS委員会の承認を経て施行する。
