## 1. プロジェクト立ち上げ
<!-- section_id: sec_1 -->

株式会社サンプルにおけるプロジェクトは、社内変革・新規事業・大型受注案件・システム導入等、定常業務の枠を超える取り組みを指します。立ち上げ段階の質がプロジェクト全体の成否を左右するため、必ず本章のプロセスに従ってください。

### 1.1 プロジェクト分類と承認権限

| 規模区分 | 予算 | 期間 | 承認 |
| :--- | :--- | :--- | :--- |
| S(小) | 500万円未満 | 3ヶ月以内 | 部長 |
| M(中) | 500万円〜3,000万円 | 3ヶ月〜1年 | 本部長+CFO |
| L(大) | 3,000万円〜1億円 | 1年超 | 経営会議 |
| XL(全社) | 1億円超 | 1年超 | 取締役会 |

### 1.2 プロジェクト憲章(Project Charter)

立ち上げ承認のためにプロジェクト憲章(Notion テンプレ:PRJ-CHARTER-01)を作成します。

| 項目 | 内容 |
| :--- | :--- |
| プロジェクト名 | プロジェクトコード(PRJ-YYYY-XXX)と通称 |
| 背景・目的 | 経営課題、解決すべき問題、目指す姿 |
| スコープ(In/Out) | 含む範囲・含まない範囲を明確化 |
| 成功基準(KPI) | 定量目標(売上、コスト削減、利用率等) |
| ステークホルダー | スポンサー、PM、コアメンバー、関連部門、顧客 |
| 概算予算 | 人件費、外注費、ライセンス費、その他 |
| 概算スケジュール | 主要マイルストーン日付 |
| 主要リスク | 立ち上げ時に想定される高リスク3件以上 |
| 前提条件・制約事項 | 法令、社内規程、リソース制約 |

### 1.3 ステークホルダー登録

ステークホルダー登録簿(Notion DB:PRJ-STAKEHOLDER-01)に以下を記録します。

| 項目 | 内容 |
| :--- | :--- |
| 氏名 | - |
| 所属・役職 | - |
| 役割 | スポンサー/PM/コア/関係者/顧客 |
| 関心領域 | 予算/スケジュール/品質/法令遵守 等 |
| 影響度(高/中/低) | プロジェクトへの影響力 |
| 関与頻度 | 週次/隔週/月次/節目 |
| 連絡方法 | 対面/Slack/メール |

### 1.4 プロジェクト審査会

M以上のプロジェクトは **プロジェクト審査会(PMO主催、毎月第2火曜開催)** に上程し、立ち上げ承認を取得します。提出書類は次の通り:

1. プロジェクト憲章(PDF)
2. 概算工数積算(Excel)
3. スケジュール案(ガントチャート)
4. ステークホルダー登録簿
5. 主要リスクリスト

承認結果は、(1)承認、(2)条件付き承認、(3)再提出、(4)却下 の4区分で通知されます。

### 1.5 キックオフミーティング

承認後、原則 **2週間以内** にキックオフを実施します。

**標準アジェンダ(90分)**:

| 時間 | 内容 |
| :--- | :--- |
| 0:00〜0:10 | スポンサー挨拶・背景説明 |
| 0:10〜0:30 | プロジェクト憲章共有 |
| 0:30〜0:50 | 体制・役割分担 |
| 0:50〜1:10 | スケジュール・マイルストーン |
| 1:10〜1:25 | 主要リスク・運営ルール |
| 1:25〜1:30 | Q&A、次回定例案内 |

> **注意:** キックオフ議事録は **当日中** に参加者へ共有してください。役割認識のズレを早期に検知することが目的です。

## 2. 計画策定(WBS)
<!-- section_id: sec_2 -->

WBS(Work Breakdown Structure)はプロジェクトの羅針盤です。当社では **Asana(主)** または **Jira(開発系)** で管理し、ガントチャート表示は Asana タイムライン機能を利用します。

### 2.1 WBSの作成原則

1. **成果物指向**: タスクではなく成果物(Deliverable)を起点に分解する。
2. **100%ルール**: WBSはプロジェクトの全成果物を網羅し、重複なく分割する。
3. **8/80ルール**: 最小タスクは8時間以上、80時間(2週間)以内を目安に分解する。
4. **担当者の特定**: すべてのタスクに **単一の担当者** を割当てる(共同責任は責任不在を生む)。
5. **完了基準の明確化**: 各タスクに「完了条件(DoD: Definition of Done)」を記述。

### 2.2 WBSテンプレ階層

```
Lv1: プロジェクト
 Lv2: フェーズ(要件定義、設計、構築、テスト、リリース)
  Lv3: ワークパッケージ(機能単位、ドキュメント単位)
   Lv4: タスク(実作業、8〜80時間)
```

### 2.3 マイルストーン

主要な意思決定ポイントや成果物完成日を **マイルストーン(MS)** として明示。

| MS番号 | 名称 | 想定日 | 判定基準 |
| :--- | :--- | :--- | :--- |
| MS1 | 要件定義完了 | YYYY/MM/DD | 要件定義書のステアリング承認 |
| MS2 | 設計完了 | YYYY/MM/DD | 設計書レビュー完了 |
| MS3 | 構築完了 | YYYY/MM/DD | 全機能の単体テスト完了 |
| MS4 | UAT完了 | YYYY/MM/DD | 顧客受入テスト合格 |
| MS5 | 本番リリース | YYYY/MM/DD | 本番環境稼働、運用引継完了 |

### 2.4 リソース計画

| 項目 | 内容 |
| :--- | :--- |
| 人的リソース表 | 氏名・役割・稼働率(0〜100%)・期間 |
| 外部リソース | 外注先・契約形態・単価・期間 |
| 設備・環境 | 開発環境、検証環境、本番環境 |
| ライセンス | SaaS、ツール、有償ライブラリ |

リソース調整は **毎週金曜** にPMがアップデートし、PMOへ共有します。

### 2.5 ガントチャート(Asana タイムライン)

1. Asanaのプロジェクトを開き、上部 **[タイムライン]** タブをクリック。
2. WBSタスクが横棒で表示される。
3. タスクをドラッグして期日変更、依存関係(矢印)を設定。
4. クリティカルパス(全体納期を決定する経路)を色分け表示。
5. リソース過負荷(担当者の同時タスクが3件超)はアラート表示。

### 2.6 計画書の承認

WBS・スケジュール・リソース計画完成後、以下に承認依頼します。

| 規模 | 承認者 |
| :--- | :--- |
| S | 部長 |
| M | 本部長+PMO |
| L | 経営会議+PMO |
| XL | 取締役会+PMO |

> **NG例:** 担当者を明示せず「チーム全員でやる」とWBSに書く → 誰も着手せずタスク放置。
>
> **OK例:** 各タスクに1名の責任者を明示、サポート要員は別カラム「協力者」に列挙。

## 3. 進捗管理
<!-- section_id: sec_3 -->

進捗管理の目的は「現状把握 → 差異分析 → 是正アクション」を高速に回すことです。当社の標準は **週次定例(60分)+随時報告** です。

### 3.1 週次定例フォーマット

| 時間 | アジェンダ | 担当 |
| :--- | :--- | :--- |
| 0:00〜0:05 | 前回アクション確認 | PM |
| 0:05〜0:20 | 進捗報告(チームリーダー別) | 各リーダー |
| 0:20〜0:30 | 全体RAG評価・主要KPI | PM |
| 0:30〜0:45 | リスク・課題ディスカッション | 全員 |
| 0:45〜0:55 | 次週計画と意思決定事項 | PM |
| 0:55〜1:00 | アクション項目確認・次回案内 | PM |

### 3.2 RAGステータス

プロジェクト全体および各ワークストリームを **RAG** で評価します。

| 区分 | 意味 | 基準例 | アクション |
| :--- | :--- | :--- | :--- |
| **Green** | 順調 | スケジュール+5%以内、予算+5%以内 | 維持 |
| **Amber** | 注意 | 遅延5〜15%、予算超過5〜15% | 是正計画策定 |
| **Red** | 危険 | 遅延15%超、予算超過15%超、重大リスク発現 | エスカレ+リカバリ計画 |

### 3.3 進捗率の算定方法

| 方式 | 算定 | 適用 |
| :--- | :--- | :--- |
| 0/100法 | 完了0% or 100% | 短期タスク |
| 50/50法 | 着手50%、完了100% | 中期タスク |
| 出来高比例 | 成果物の出来高率 | 開発、制作 |
| マイルストーン法 | MS到達数/全MS数 | 大規模プロジェクト |

### 3.4 EVM(簡易)

予算管理にはEVM(Earned Value Management)の簡易版を使用します。

| 指標 | 算式 | 意味 |
| :--- | :--- | :--- |
| PV(計画価値) | 計画工数 × 単価 | 計画上当該日までに完了すべき金額 |
| EV(出来高) | 実完了率 × 計画予算 | 実際に完了した分の金額換算 |
| AC(実コスト) | 実際の支出 | 当該日までの実支出 |
| SV(スケジュール差異) | EV - PV | プラス:先行、マイナス:遅延 |
| CV(コスト差異) | EV - AC | プラス:節約、マイナス:超過 |
| SPI | EV / PV | 1超:先行、1未満:遅延 |
| CPI | EV / AC | 1超:節約、1未満:超過 |

> **注意:** EVMは目的ではなく手段です。SPI=0.9なら「予定の9割しか進んでいない」と直感的に解釈し、是正策に時間を使ってください。

### 3.5 進捗報告のレポートライン

| 報告先 | 頻度 | 形式 |
| :--- | :--- | :--- |
| PM内部 | 日次 | Slack #prj-XXX 朝会 |
| プロジェクトチーム | 週次 | 週次定例 |
| 部長・PMO | 隔週 | ステアリング(30分) |
| 本部長・経営層 | 月次 | 月次レビュー(60分) |
| 顧客(該当時) | 月次 | 顧客報告会 |

### 3.6 遅延発生時の是正

1. 遅延を認識した時点で速やかに **遅延報告(Slack + メール)**。
2. 24時間以内に **是正計画(Notion テンプレ:PRJ-RECOVERY-01)** を作成。
   - 原因分析(なぜなぜ5回)
   - 是正策(複数案を比較)
   - 影響範囲(スコープ・コスト・品質・スケジュールの再評価)
   - 必要追加リソース
3. 是正計画は **PMO+スポンサー** の承認を得て実行。

## 4. リスク・課題管理
<!-- section_id: sec_4 -->

リスク(発生する可能性のある事象)と課題(既に発生している事象)を区別し、双方を体系的に管理します。

### 4.1 リスク登録簿

リスクは **リスク登録簿(Notion DB:PRJ-RISK-01)** に集約します。

| 項目 | 内容 |
| :--- | :--- |
| リスクID | RSK-YYYY-XXX |
| カテゴリ | スケジュール/コスト/品質/リソース/外部/法令/技術 |
| リスク内容 | 「もし◯◯が起きると□□となる」形式 |
| 発生確率(1〜5) | 1:稀 〜 5:ほぼ確実 |
| 影響度(1〜5) | 1:軽微 〜 5:致命的 |
| スコア(確率×影響) | 1〜25(15以上で要対応) |
| 対応戦略 | 回避/軽減/転嫁/受容 |
| 予防策 | 発生を未然に防ぐ施策 |
| コンティンジェンシープラン | 発生時の対応プラン |
| 対応責任者 | 氏名 |
| ステータス | オープン/監視中/クローズ |

### 4.2 リスクヒートマップ

| 影響度＼確率 | 1(稀) | 2 | 3 | 4 | 5(確実) |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **5(致命)** | 5 緑 | 10 黄 | 15 赤 | 20 赤 | 25 赤 |
| **4** | 4 緑 | 8 黄 | 12 黄 | 16 赤 | 20 赤 |
| **3** | 3 緑 | 6 緑 | 9 黄 | 12 黄 | 15 赤 |
| **2** | 2 緑 | 4 緑 | 6 緑 | 8 黄 | 10 黄 |
| **1(軽微)** | 1 緑 | 2 緑 | 3 緑 | 4 緑 | 5 緑 |

赤(15以上)は即時の予防策・コンティンジェンシープランが必須。黄(8〜14)は予防策のみ準備。緑(7以下)はモニタのみ。

### 4.3 課題台帳

実際に発生した問題は **課題台帳(Notion DB:PRJ-ISSUE-01)** で管理。

| 項目 | 内容 |
| :--- | :--- |
| 課題ID | ISS-YYYY-XXX |
| 発見日 | YYYY/MM/DD |
| 発見者 | 氏名 |
| 内容 | 事実ベースで記述 |
| 影響範囲 | スコープ/コスト/品質/スケジュール |
| 優先度 | 緊急/高/中/低 |
| 担当者 | 氏名 |
| 期限 | YYYY/MM/DD |
| ステータス | 起票/着手/解決待ち/解決済 |
| 解決策・経緯 | 対応した内容 |

### 4.4 課題エスカレーション

| 優先度 | エスカ先 | 期限 |
| :--- | :--- | :--- |
| 緊急 | PM→部長 即時、本部長 24時間以内 | - |
| 高 | PM→部長 当日中 | 3営業日 |
| 中 | PM 週次定例で報告 | 2週間 |
| 低 | PM 月次でレビュー | 1ヶ月 |

> **NG例:** 課題を発見しても「もう少し様子を見よう」と起票しない → 後で大きな問題に発展。
>
> **OK例:** 課題発見即時に起票し、優先度判定はPMに任せる。起票自体に価値がある。

### 4.5 リスク・課題レビュー

週次定例の中で **5〜10分** をリスク・課題レビューに充てます。新規追加、ステータス更新、クローズ判断を行います。

## 5. 完了報告
<!-- section_id: sec_5 -->

プロジェクト完了は単なる「終了」ではなく、組織学習の起点です。クロージング・振り返り・ナレッジ化を必ず実施してください。

### 5.1 完了条件(DoD)

| 観点 | 完了条件 |
| :--- | :--- |
| 成果物 | 全成果物を顧客・社内ステークホルダーが受領済 |
| 品質 | 受入テスト合格、未解決クリティカル不具合ゼロ |
| 引継ぎ | 運用部門への引継書を提出、運用部門が了承 |
| 契約 | 契約期間内の義務完了、検収完了 |
| 財務 | 全請求・支払処理完了、未払金なし |
| ドキュメント | 完了報告書、ナレッジ記事を作成 |

### 5.2 完了報告書

**Notion テンプレ:PRJ-CLOSING-01** を使用します。

| 項目 | 内容 |
| :--- | :--- |
| プロジェクト名・コード | - |
| 期間 | 計画 vs 実績 |
| 予算 | 計画 vs 実績(差異分析) |
| スコープ | 計画 vs 実績(変更履歴) |
| KPI達成度 | 計画 vs 実績(達成率) |
| 主要成果物 | リスト+格納場所(Box URL) |
| 主要リスク・課題 | 発生したものと対応内容 |
| 顧客満足度(該当時) | NPS/CSATスコア |
| 教訓(Lessons Learned) | KPT形式で記載 |

### 5.3 KPTによる振り返り

完了報告会の冒頭でKPT(Keep / Problem / Try)を実施。

| 観点 | 内容 | 件数目安 |
| :--- | :--- | :--- |
| Keep(良かったこと) | 継続すべき行動・取り組み | 5件以上 |
| Problem(問題) | 改善すべき事象・原因 | 3件以上 |
| Try(次回試すこと) | 具体的な改善アクション | 3件以上 |

参加者全員が **付箋(Miro/Mural等)** に書き出し、グルーピング→投票→アクションオーナー指名。

### 5.4 ナレッジ化

教訓・成果物・テンプレートを **社内ナレッジベース(Notion > PMO > Project Archive)** に登録します。

1. 完了報告書PDF
2. 主要設計書・成果物の最終版
3. ベストプラクティス記事(Try項目を一般化)
4. 次回類似プロジェクトに残すテンプレ・チェックリスト

### 5.5 メンバー解散と評価

1. プロジェクトメンバーは正式に解散通知(Slack #prj-XXX で告知)。
2. PMがメンバー1名ずつに **個別フィードバック** を実施(30分)。
3. メンバーの評価は元所属上長へ書面共有(人事システムに記録)。
4. プロジェクトに貢献した社員は **半期表彰** の候補に推薦可能。

### 5.6 アーカイブ

1. Slackチャンネルは **3ヶ月後にアーカイブ**(完全削除はせず参照可能)。
2. Box共有フォルダは **「Closed」フォルダ配下に移動**。
3. Asana/Jiraプロジェクトは **「Archived」ステータス** に変更。
4. 契約関連書類は契約管理規程(doc_015)に従い保管。

> **注意:** プロジェクトの「未完成」「中止」「途中解散」も正式な完了プロセスとして扱います。中止理由・経緯を必ず文書化し、組織知として残してください。「黒歴史」として隠さないことが学習組織の文化です。

> **FAQ:** Q. 完了報告会には誰を呼ぶべきですか?  A. スポンサー、PM、コアメンバー、関連部門代表、運用部門代表、PMO は必須。任意で社内の関連プロジェクトPMも招くと、横断学習が促進されます。
